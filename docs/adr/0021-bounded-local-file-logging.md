# ADR-0021: 有界本地文件日志与热更新策略

- 状态：Accepted
- 日期：2026-07-21
- 决策者：maintainer

## 背景

any2api 已使用 `tracing` 输出控制台日志，但尚未把 `logs.file.level`、`logs.file.retention` 与 `logs.file.max_total_size` 接入 SettingRegistry。个人单节点部署需要可直接查看的本地历史日志，同时文件写入和清理不能阻塞公开请求，也不能在配置提交后增加一个可能失败的发布步骤。

## 决策

- 本地文件日志写入 `<data-dir>/logs` 下的 JSONL 分段文件；控制台继续由 `RUST_LOG` 控制，`logs.file.level` 只控制文件层。
- 文件层使用有界、丢弃式非阻塞队列和独立写线程。数据面不等待磁盘；进程结束时由 Guard 尽力刷新已经入队的事件。
- 文件按 UTC 日期和大小分段。单段目标上限取 `logs.file.max_total_size / 8` 与 `32 MiB` 的较小值，使默认配置能保留多个分段，并避免当前活跃文件无限突破目录容量。
- 清理只处理 any2api 自己命名的 `.jsonl` 文件，先删除超过 `logs.file.retention` 的关闭分段，再从最旧分段开始删除，直到为当前分段预留空间后不超过 `logs.file.max_total_size`。活跃文件不在清理中删除；单个完整日志事件可以使分段短暂超过目标上限。
- 三项设置继续由同一 SettingRegistry、SQLite 覆盖值和 ConfigPublisher 发布。配置提交后的 reconcile 只更新内存级别与策略，不执行文件 I/O，也不返回错误。
- 日志级别在 reconcile 后立即影响新事件；保留与容量策略由写线程在启动、策略变化后的下一次合格写入以及后续轮转时应用。清理失败只使文件日志写入计为失败，不回滚配置或阻塞网关。
- 本地文件日志不保存或恢复请求运行状态，也不建立新的配置文件、备份系统或远程日志服务。

## 备选方案

- 只使用每日轮转：活跃日志文件可能无限增长，无法兑现总容量上限，放弃。
- 在 ConfigPublisher 中同步扫描和删除文件：会把磁盘 I/O 放入 Commit 后无失败发布段，破坏原子发布边界，放弃。
- 引入通用日志框架、远程采集或压缩归档：超出当前个人单节点需求，增加了不必要的依赖和恢复语义，放弃。

## 后果

日志写入不会对请求施加磁盘背压；极端写入压力下允许丢弃文件日志。容量上限以完整事件和分段为边界，不承诺逐字节硬实时精确。修改保留期或容量后，不需要重启，但磁盘清理发生在文件线程下一次工作时。

## 验证

- Domain 测试覆盖三项设置的默认值、类型、范围和枚举契约。
- 文件写入测试覆盖 JSONL 写入、大小/日期轮转、按期限清理、按总容量从最旧文件清理和不删除非托管文件。
- Publisher 测试覆盖同一 revision 下 RequestTelemetry 与文件日志策略同步 reconcile。
- 管理 API 与 Web 测试覆盖三项设置的默认值、覆盖值、热更新和恢复默认。
- 提交前运行 Rust、Web 与架构门禁。
