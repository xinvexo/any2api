# ADR-0002: Secret Vault 主密钥与版本化 AEAD

- 状态：Accepted
- 日期：2026-07-18
- 决策者：maintainer

## 背景

any2api 后续需要持久化 Provider API Key、OAuth Token 和代理密码。SQLite 必须只保存密文，主密钥位于数据库外；同时项目面向个人自托管，首次启动不能要求用户手工生成复杂密钥，但已有密文时也绝不能因文件丢失而静默换钥。

## 决策

- 首版使用 XChaCha20-Poly1305，主密钥 256 位，每个信封使用 CSPRNG 生成的 192 位独立 nonce。
- 默认主密钥路径为 `<data-dir>/master-key.json`，允许用 `ANY2API_MASTER_KEY_FILE` 指向受保护文件或容器 Secret 挂载。
- 只有 SQLite 尚无 Vault 元数据时才允许首次初始化：已有文件则读取；没有文件则以 create-new 语义生成版本化文件。
- 首次初始化在 SQLite 中保存一个加密校验哨兵。之后每次启动必须用当前文件解密哨兵；文件缺失、格式错误、权限过宽、Key ID 不符、版本未知或认证失败均使启动失败。
- Secret 信封保存 `version`、`key_id`、`algorithm`、`nonce`、`ciphertext` 和 `aad_version`。业务 Secret 的 AAD 使用固定二进制编码，并至少绑定记录 ID、Secret 类型和 Provider 类型。
- Unix 创建权限为 `0600`，读取时拒绝 group/other 权限。Windows 依赖数据目录或挂载文件继承的用户 DACL，部署文档必须要求限制其他账户读取。
- 首版不实现在线主密钥轮换、KMS、内建备份或恢复。迁移主密钥必须在未来单独设计可验证的离线流程。

## 备选方案

- 每次缺失时自动生成：已有密文会永久不可解密，放弃。
- 只允许环境变量直接传入密钥：容易进入进程环境和部署日志，也增加本地使用门槛；首版改用文件路径覆盖。
- AES-GCM：可以满足要求，但 XChaCha20 的大 nonce 空间更适合随机 nonce 与本地单节点实现，且不依赖 AES 硬件。
- 把主密钥放进 SQLite：数据库副本会同时包含密文和解密材料，放弃。

## 后果

首次运行会在数据目录创建一个必须单独保护的主密钥文件；复制 SQLite 而遗漏该文件无法恢复 Secret。反过来，只有主密钥文件而没有数据库也无法恢复配置。启动阶段会验证文件权限、格式和数据库校验哨兵，安全失败会阻止服务监听。

## 验证

- 单元测试覆盖加解密往返、随机 nonce、错误 AAD、篡改、未知版本和脱敏 Debug。
- Storage 集成测试覆盖首次生成、已有文件初始化、重启复用、缺失文件、错误密钥和不安全 Unix 权限。
- 启动契约测试确认 Vault 校验发生在读取配置和监听端口之前。
