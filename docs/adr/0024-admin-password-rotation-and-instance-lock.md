# ADR-0024: 管理员密码在线轮换与数据目录单实例锁

- 状态：Accepted
- 日期：2026-07-22
- 决策者：maintainer

## 背景

单管理员认证已经具备 Argon2id、服务端会话和 CSRF，但管理员只能通过首次 Setup 或重启环境变量初始化密码。SQLite 中的摘要、进程内缓存和会话状态还没有在线变更契约；同时架构要求单节点数据目录只能由一个进程使用，启动边界尚未实际取得实例锁。

## 决策

- 新增受保护的 `POST /api/admin/auth/password/rotate`。请求体只包含 `current_password` 和 `new_password`，沿用现有 Session + CSRF 中间件。
- `AdminCredentialStore` 增加基于旧摘要的 CAS 替换端口；SQLite 使用单条 `UPDATE ... WHERE singleton = 1 AND password_hash = ?`，并更新 `updated_at`。不新增 migration。
- `AdminAuthService` 使用凭据写锁和摘要 `RwLock`：登录在摘要验证到签发会话之间持有读锁，轮换持有写锁完成验证、哈希、CAS、内存替换和会话重建。轮换操作在独立 Tokio 任务中执行，调用方取消不会中断已经开始的变更。
- 轮换成功后清空当前进程全部旧会话和登录失败窗口，只签发当前请求的新 Cookie/CSRF。旧 Cookie 立即失效，其他浏览器重新登录。
- 当前密码错误使用 `403 admin_current_password_invalid`；它不能触发 Web 客户端对受保护 `401` 的全局会话过期处理。新密码格式错误仍使用 `400 admin_invalid_password`。
- 数据库 CAS 未更新时不以本次新摘要覆盖持久状态；服务重新加载当前持久摘要、清理现有会话并返回 `409 admin_password_changed`，要求管理员重新登录后重试。
- `ANY2API_ADMIN_PASSWORD` 保持仅缺失初始化语义。
- 启动时在数据目录创建并独占锁定 `any2api.instance.lock`，成功取得后才打开 SQLite；锁对象存活至 `run` 返回，获取失败直接退出。该锁是单节点内存状态正确性的前置，不做跨进程会话同步。

## 取舍

- 不增加密码版本表、事件总线或 Redis。单管理员、单节点场景只需要摘要 CAS 和进程内锁。
- 不保留旧会话等待自然过期；轮换是明确的凭据边界，立即撤销更容易理解和验证。
- 不把密码轮换加入配置 revision；管理员认证状态独立于数据面 PublishedSnapshot。

## 后果

- 当前浏览器无需被强制登出，轮换成功后直接获得新会话；其他已登录浏览器会收到 401 并回到登录页。
- 客户端断开后，已经启动的轮换仍会完成，可能导致浏览器仍持有已失效的旧 Cookie；这是可接受的，重新登录即可。
- 实例锁文件属于运行时控制文件，不是配置导出或备份内容；进程退出时释放系统锁，文件本身可以保留。
- 未实现内建 TLS、密码跨重启会话恢复或 OAuth2；这些仍按独立 ADR/阶段推进。

## 验证

- Storage：CAS 成功、旧摘要不匹配和重启读取。
- Server：当前密码错误不改摘要、全会话撤销、当前会话重签发、并发旧密码登录不能跨轮换成功。
- HTTP：CSRF、错误状态码、旧 Cookie 失效、新 Cookie/新密码生效、重启后新摘要生效。
- Web：表单只在局部状态保存密码，成功清空输入并更新 CSRF；当前密码错误不触发全局会话过期事件。
- App：同一数据目录第二个实例无法取得锁，锁释放后可以重新取得。
